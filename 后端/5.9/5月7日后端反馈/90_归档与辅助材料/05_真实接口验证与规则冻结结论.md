## 二、冻结结论 1：task_queue -> tasks 状态机

### 当前实现结论

- `POST /api/v1/tasks` 会同步写入：
  - `tasks`
  - `task_assignments`
  - `task_updates`
  - `task_queue`
  - `audit_logs`
- `GET /api/v1/tasks` 当前主读源为 `tasks`
- `task_queue` 当前仅保留兼容镜像职责

### 冻结口径

从 2026-05-07 起，统一按以下规则解释：

1. `tasks` 是正式任务主表
2. `task_assignments` 是任务归属关系表
3. `task_updates` 是任务状态/事件流水表
4. `task_queue` 是兼容层/镜像表
5. `GET /api/v1/tasks` 以后统一以 `tasks` 为权威读源

### 状态流转

建议冻结为：

1. `pending`
2. `in_progress`
3. `done`
4. `closed`

补充规则：
- 创建任务默认 `pending`
- 开始执行进入 `in_progress`
- 完成进入 `done`
- 取消、归档或关闭进入 `closed`

### 给全栈的解释口径

如果前端再问“现在到底看 `task_queue` 还是 `tasks`”，统一回复：

当前真实接口查询已经以 `tasks` 为主，`task_queue` 只保留兼容镜像用途，不再作为正式任务读源。

## 三、冻结结论 2：日报双写主从规则

### 当前实现结论

当前项目内存在两层日报落库：

1. 接口快照层：
   - `POST /api/v1/report/staff`
   - 面向联调输出 `staff_reports`

2. 结构化持久层：
   - 内部结构化落库到 `work_records + work_record_items`
   - 用于后续模板化、分析和正式数据沉淀

### 冻结口径

从 2026-05-07 起，统一按以下规则解释：

1. `staff_reports` 是 Staff 日报快照/展示层
2. `work_records` 是结构化日报主记录
3. `work_record_items` 是结构化日报字段明细
4. 前端/Agent 常规联调以 `staff_reports` 接口响应口径为准
5. 后端结构化沉淀与后续分析以 `work_records/work_record_items` 为准

### 主从关系冻结

- 面向联调与展示：
  - 主读源：`staff_reports`
- 面向结构化沉淀：
  - 主写源：`work_records + work_record_items`
- `staff_reports` 保留为接口快照与展示快照

### 给全栈的解释口径

如果前端再问“日报到底写哪张表”，统一回复：

前端和 Agent 不直接面向数据库表，当前只需要对接：
- `POST /api/v1/report/staff`
- `GET /api/v1/report/staff`

后端内部按“快照层 + 结构化层”双轨处理。
