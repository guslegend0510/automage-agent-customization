## 一、当前后端已确认的真实口径

### 1. Staff 日报链路

当前后端已真实跑通：

1. `POST /api/v1/report/staff`
2. `GET /api/v1/report/staff`

当前统一口径：

1. 前端联调入口以：
   - `POST /api/v1/report/staff`
   - `GET /api/v1/report/staff`
   为准
2. `staff_reports` 承担快照/展示层职责
3. `work_records + work_record_items` 承担结构化沉淀职责
4. 这不是二选一，而是“接口快照层 + 结构化持久层”双轨

给文档 Owner 的反馈：

- 文档里可以继续保留 `staff_reports` 与 `work_records/work_record_items` 的分层说明
- 但必须补一句：当前真实联调以 `POST/GET /api/v1/report/staff` 返回口径为准，不要求前端直接理解数据库表结构

### 2. Manager 汇总链路

当前后端已真实跑通：

1. `POST /api/v1/report/manager`
2. `GET /api/v1/report/manager`

当前统一口径：

1. `manager_reports` 是当前联调快照主落点
2. `summaries + summary_source_links` 是后端内部归并、溯源、扩展层
3. 当前前端联调读取口径应以 `manager_reports` 快照能力为主

给文档 Owner 的反馈：

- 文档可以保留 `manager_reports + summaries + summary_source_links` 的分层设计
- 但不要再写成“前端需要理解内部归并表关系”
- 当前最小真实联调，应明确以 `POST/GET /api/v1/report/manager` 为准

### 3. Executive 决策链路

当前后端已真实跑通：

1. `POST /api/v1/decision/commit`

当前统一口径：

1. `decision_records` 是正式决策主记录
2. `decision_logs` 是正式动作流水
3. `agent_decision_logs` 更偏向 Agent 推理/生成过程日志

给文档 Owner 的反馈：

- 文档把决策主落点从临时 `summaries.meta` 收敛到 `decision_records + decision_logs` 的方向是对的
- 当前正式口径应明确朝这个结构收口

### 4. 任务链路

当前后端已真实跑通：

1. `POST /api/v1/tasks`
2. `PATCH /api/v1/tasks/{task_id}`
3. `GET /api/v1/tasks`

当前统一口径：

1. 正式任务主域是：
   - `tasks`
   - `task_assignments`
   - `task_updates`
2. `task_queue` 只保留兼容镜像语义
3. 当前 `PATCH /api/v1/tasks/{task_id}` 已支持按业务 `task_id` 更新

给文档 Owner 的反馈：

- “老板确认后正式任务进入 `tasks`” 这条说法可以保留
- “task_queue 可理解为草稿/队列兼容层” 也可以保留
- 但必须补一句：当前真实接口查询任务已经统一以 `tasks` 为主，不再以 `task_queue` 为正式读源

### 5. 审计与权限

当前后端已具备：

1. 基础 Bearer Token / 角色鉴权能力
2. 关键接口角色边界校验
3. 任务查询隔离与 manager 部门视角过滤
4. 关键写接口审计写入能力

给文档 Owner 的反馈：

- 文档里“复杂完整权限矩阵后续再做”的表述可以保留
- 但不能再写成“当前没有真实权限能力”
- 更准确的写法应是：
  - 已具备基础 RBAC 与关键接口边界控制
  - 更完整的策略引擎仍属于后续增强项

## 二、这套文档仍需修正的点

### 1. `task_queue` 的表述需要回收旧口径

当前问题：

- 文档里仍多处把 `task_queue` 写成任务主链路核心对象

建议改法：

1. 保留“任务草稿 / 队列 / 兼容镜像”的描述
2. 删除或弱化“当前任务主链路以 `task_queue` 为准”的旧说法
3. 统一改成：
   - 正式任务主表：`tasks`
   - 归属关系表：`task_assignments`
   - 状态流水表：`task_updates`
   - 兼容镜像层：`task_queue`

### 2. 节点 ID 示例口径需要统一

当前问题：

- 历史文档里还混有旧示例节点
- 当前真实联调文档已统一采用：
  - `staff_agent_mvp_001`
  - `manager_agent_mvp_001`
  - `executive_agent_boss_001`

建议改法：

1. 整份里程碑二文档统一成一套 `node_id / manager_node_id`
2. 如果保留旧示例，必须标注为历史样例，不作为当前联调主口径

### 3. 用户 ID 口径需要显式区分

当前问题：

- 文档中同时出现业务账号和数据库数字 ID
- 真实环境里还存在“文档示例账号”和“本地真实种子账号”两套口径

建议改法：

1. 显式区分：
   - 业务 `user_id`
   - 数据库主键 `record_id / numeric id`
2. Mock 场景尽量统一使用业务账号
3. 只有数据库主键语境才使用数字 ID

### 4. `POST /api/v1/incidents` 不能写成当前最小真实链路必经项

当前问题：

- 文档里把高风险日报和 `POST /api/v1/incidents` 绑定得过强

建议改法：

1. 可保留为可选扩展接口
2. 不要写成当前最小真实联调必测项
3. 如果继续保留，应明确标注：
   - “本轮未作为最小真实联调必测项”

### 5. “完整权限未实现”的旧说法需要更新

当前问题：

- 这类描述在早期 Mock 阶段成立
- 但放到 2026-05-07 当前后端状态已经不准确

建议改法：

1. 改成“完整策略引擎未实现”
2. 保留“复杂权限矩阵后续补强”
3. 同时补充：
   - 基础 Bearer Token
   - 角色鉴权
   - 关键接口角色边界
   - 任务查询隔离
   已具备

### 6. 联调账号口径必须补充说明

当前问题：

- 文档示例里常写：
  - `user_executive_001`
  - `user_manager_001`
  - `user_agent_001`
- 但当前本地真实种子数据实际账号是：
  - `chenzong`
  - `lijingli`
  - `zhangsan`
  - `wangxiaomei`

建议改法：

1. 文档中补一段“示例账号口径”和“真实本地种子账号口径”的区别说明
2. 避免前端按旧示例账号直连当前真实本地库时误判为后端异常
