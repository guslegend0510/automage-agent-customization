# 给全栈的后端联调答复

当前后端主链路已经能联调，真正需要先统一的不是接口协议，而是联调口径。现在最大的风险是把“旧 Mock 示例账号”和“当前本地真实库账号”混着用，导致前端误以为后端接口有问题。

如果你们现在打的是当前本地真实环境 `http://localhost:8000`，请统一按下面这套走：

- `org_id=org_automage_mvp`
- `department_id / dept_id=dept_mvp_core`
- Staff `user_id=zhangsan`
- Manager `user_id=lijingli`
- Executive `user_id=chenzong`
- Staff `node_id=staff_agent_mvp_001`
- Manager `node_id=manager_agent_mvp_001`
- Executive `node_id=executive_agent_boss_001`

不要再直接使用旧文档里的：

- `user_agent_001`
- `user_backend_001`
- `user_manager_001`
- `user_executive_001`

这些账号现在只适合 Mock/测试说明，不适合直接打当前真实本地库。

当前后端统一接口口径是：

1. Staff 日报只按：
   - `POST /api/v1/report/staff`
   - `GET /api/v1/report/staff`
   联调
2. Manager 汇总只按：
   - `POST /api/v1/report/manager`
   - `GET /api/v1/report/manager`
   联调
3. Task 链路只按：
   - `POST /api/v1/tasks`
   - `PATCH /api/v1/tasks/{task_id}`
   - `GET /api/v1/tasks`
   联调
4. 当前正式任务主域是：
   - `tasks`
   - `task_assignments`
   - `task_updates`
5. `task_queue` 只保留兼容镜像语义，不再作为正式任务主读源

建议前端按下面顺序联调：

1. `POST /api/v1/report/staff`
2. `GET /api/v1/report/staff`
3. `POST /api/v1/report/manager`
4. `GET /api/v1/report/manager`
5. `POST /api/v1/tasks`
6. `PATCH /api/v1/tasks/{task_id}`
7. `GET /api/v1/tasks`

一句话结论：

当前后端不是接口没打通，而是联调口径必须统一；只要当前真实环境统一使用 `zhangsan / lijingli / chenzong + org_automage_mvp + dept_mvp_core + staff_agent_mvp_001 / manager_agent_mvp_001 / executive_agent_boss_001` 这套身份参数，主链路就可以正常联调。
