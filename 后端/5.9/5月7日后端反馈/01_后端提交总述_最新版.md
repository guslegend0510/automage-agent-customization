## 二、本轮已完成的后端收口项

### 1. 真实接口闭环

本轮已完成以下接口的真实验证与闭环打通：

- `POST /api/v1/report/staff`
- `GET /api/v1/report/staff`
- `POST /api/v1/report/manager`
- `GET /api/v1/report/manager`
- `POST /internal/dream/run`
- `POST /api/v1/decision/commit`
- `POST /api/v1/tasks`
- `GET /api/v1/tasks`
- `PATCH /api/v1/tasks/{task_id}`

其中本轮重点补实的是经理汇总到任务状态流转链路，即：

1. 经理提交汇总日报。
2. Dream 生成决策草稿。
3. 高层提交正式决策。
4. 决策下发正式任务。
5. 员工查询并更新任务。
6. 审计轨迹与任务更新记录同步落库。

### 2. 幂等策略落地

本轮已不再停留在“接口约定层”的幂等，而是补到了真实行为层：

- Staff 日报：
  相同 `org_id + user_id + record_date` 且内容一致时直接返回既有记录；内容不一致时返回 `409 Conflict`。
- 任务创建：
  相同 `task_id` 且内容一致时直接返回既有任务；内容不一致时返回 `409 Conflict`。
- 任务更新：
  相同 `task_id + request_id` 且内容一致时直接返回当前任务且不重复落库；内容不一致时返回 `409 Conflict`。

### 3. 数据库约束落库

本轮已将关键幂等约束真正落到数据库层：

- `work_records(org_id, user_id, record_date)` 有效记录唯一约束已落库。
- `task_updates.request_id` 已补成正式字段。
- `task_updates(request_id)` 查询索引已落库。
- `task_updates(task_id, request_id)` 唯一索引已落库。

因此当前状态已经不是“只有服务层判断”，而是“服务层判断 + 数据库约束”双保险。

### 4. 权限与隔离规则补实

本轮已进一步补实并验证以下 RBAC 规则：

- 只有 `executive` 可以调用 `POST /internal/dream/run` 与 `POST /api/v1/decision/commit`
- `manager` 不能跨部门提交汇总或跨部门建任务
- `manager` 查询经理汇总时只可看到本人/本部门范围内数据
- `staff` 查询任务时会被强制收窄到本人可见范围
- `staff` 不能越权更新非本人、非本职责范围任务

### 5. 冲突返回体统一

以下 4 类冲突场景已统一成稳定中文返回结构：

- Staff 日报冲突
- 正式决策提交冲突
- 任务创建冲突
- 任务更新冲突

统一返回体结构如下：

```json
{
  "code": 409,
  "data": null,
  "msg": "任务更新冲突",
  "error": {
    "error_type": "conflict",
    "error_code": "task_update_conflict",
    "message": "同一 request_id 已用于该任务的其他更新内容，不能重复覆盖。",
    "conflict_target": "task_update:task_id+request_id",
    "request_id": "req_xxx"
  }
}
```
