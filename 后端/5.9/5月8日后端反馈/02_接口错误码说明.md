# 接口错误码说明
日期：2026-05-08

## 一、文档目的

本文档用于说明当前后端主链路相关接口的错误码语义，便于前端、测试、联调和验收阶段统一理解与处理。

---

## 二、当前错误码定义

截至 2026-05-08，当前接口主要使用以下错误码：

| 错误码 | 含义 | 典型场景 |
| --- | --- | --- |
| `403` | 无权限访问或超出授权范围 | 角色不符、跨部门访问、越权更新 |
| `404` | 目标资源不存在 | 查询或更新的对象不存在 |
| `409` | 请求冲突 | 幂等冲突、重复提交但内容不一致、重复更新冲突 |

---

## 三、`403` 说明

### 1. 角色不允许

适用场景：

- `manager` 或 `staff` 调用 `/internal/dream/run`
- `manager` 或 `staff` 调用 `POST /api/v1/decision/commit`
- `staff` 调用 `POST /api/v1/report/manager`

示例返回：

```json
{
  "detail": "Role manager is not allowed to access this resource"
}
```

### 2. 超出授权范围

适用场景：

- 经理跨部门提交汇总
- 经理跨部门创建任务
- 员工越权更新任务

示例返回：

```json
{
  "detail": "Manager report is outside your RBAC scope"
}
```

或：

```json
{
  "detail": "Task create request is outside your RBAC scope"
}
```

或：

```json
{
  "detail": "You are not allowed to update this task"
}
```

---

## 四、`404` 说明

适用场景：

- 更新不存在的任务
- 查询不存在的资源记录

示例返回：

```json
{
  "detail": "记录不存在"
}
```

---

## 五、`409` 说明

### 1. 员工日报重复提交冲突

适用场景：

- 同一员工、同一日期重复提交日报
- 且新旧内容不一致

示例返回：

```json
{
  "code": 409,
  "data": null,
  "msg": "员工日报提交冲突",
  "error": {
    "error_type": "conflict",
    "error_code": "staff_report_conflict",
    "message": "同一员工在同一日期的日报已存在，且提交内容不一致。",
    "conflict_target": "staff_report:org_id+user_id+record_date",
    "request_id": "req_xxx"
  }
}
```

### 2. 任务创建冲突

适用场景：

- 同一 `task_id` 已存在
- 但本次提交内容与既有任务不一致

示例返回：

```json
{
  "code": 409,
  "data": null,
  "msg": "任务创建冲突",
  "error": {
    "error_type": "conflict",
    "error_code": "task_create_conflict",
    "message": "任务 ID 已存在，但本次提交内容与已有任务不一致。",
    "conflict_target": "task_create:task_id",
    "request_id": "req_xxx"
  }
}
```

### 3. 正式决策提交冲突

适用场景：

- 正式决策提交触发任务候选冲突校验

示例返回：

```json
{
  "code": 409,
  "data": null,
  "msg": "正式决策提交冲突",
  "error": {
    "error_type": "conflict",
    "error_code": "decision_commit_conflict",
    "message": "正式决策提交触发了已存在任务的冲突校验，请确认任务候选是否重复。",
    "conflict_target": "decision_commit:task_candidates",
    "request_id": "req_xxx"
  }
}
```

### 4. 任务更新冲突

适用场景：

- 同一 `task_id + request_id` 被重复使用
- 且两次更新内容不一致

示例返回：

```json
{
  "code": 409,
  "data": null,
  "msg": "任务更新冲突",
  "error": {
    "error_type": "conflict",
    "error_code": "task_update_conflict",
    "message": "同一 request_id 已用于该任务的其他更新内容，不能重复覆盖。",
    "conflict_target": "task_update:task_id+request_id",
    "request_id": "req_xxx"
  }
}
```

## 六、重点接口与错误码对应关系

| 接口 | 重点错误码 | 说明 |
| --- | --- | --- |
| `POST /api/v1/report/manager` | `403` | 非授权部门提交将被拒绝 |
| `POST /api/v1/tasks` | `403 / 409` | 越权创建拒绝；重复任务冲突 |
| `PATCH /api/v1/tasks/{task_id}` | `403 / 404 / 409` | 越权更新、目标不存在、请求冲突 |
| `/internal/dream/run` | `403 / 404` | 角色不符或汇总不存在 |
| `POST /api/v1/decision/commit` | `403 / 409` | 角色不符或任务候选冲突 |

---

## 七、使用建议

1. 对 `403` 按权限问题处理，不建议直接重试。
2. 对 `404` 按资源不存在处理，建议核对目标 ID 或刷新数据。
3. 对 `409` 按冲突处理，建议保留 `error_code`、`conflict_target`、`request_id` 便于定位。
4. 接口使用方在错误提示上建议优先读取：
   - `detail`
   - `error.message`
   - `error.error_code`

---

## 八、结论

当前后端错误码语义已经统一，可支持联调、测试和验收阶段的标准化处理。
