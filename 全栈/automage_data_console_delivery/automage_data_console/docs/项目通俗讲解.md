# AutoMage-2 数据中台 — 通俗讲解

## 这个项目是什么

AutoMage-2 是一个 **AI 驱动的企业组织管理控制台**。把公司的日常运转流程——员工写日报 → 经理汇总 → 老板做决策 → 分配任务 → 员工执行——全部数字化，并在关键环节引入 AI Agent 来辅助分析和决策。

系统内有**三种身份**，对应公司三层人：

| 身份 | 角色 | 当前 Demo 用户 |
|------|------|----------------|
| Staff | 一线员工 | 张三 (zhangsan) |
| Manager | 部门经理 | 李经理 (lijingli) |
| Executive | 高管/老板 | 陈总 (chenzong) |

每个身份看到的页面不同、能做的操作不同、数据访问范围也不同（RBAC 权限隔离）。

## 核心业务流程（完整闭环）

整条链路在代码里定义为 21 个 Workflow 节点 (`src/data/demoWorkflow.ts`)，下面按阶段来讲。

### 一、运行上下文准备

系统启动后，加载运行环境（组织 ID、部门 ID、当前日期），初始化各角色的 Agent 会话 (`POST /api/v1/agent/init`)。

### 二、Staff 日报录入

1. **员工填写日报**：张三在 Staff Workspace 里写今天干了什么、遇到什么问题、明天计划干什么
2. **Staff Agent 整理**：系统内的 L1 AI Agent 把自然语言日报整理成结构化数据（提取关键指标、风险项、待办事项等）
3. **提交到后端**：调用 `POST /api/v1/report/staff`

### 三、后端校验与落库

1. **Schema 校验**：确认 JSON 结构合法、字段齐全
2. **冲突检测**：同一个员工同一天不能提交重复且内容不同的日报（幂等保护）
3. **数据落库**：写入 `work_records`（工作记录）、`work_record_items`（工作条目）、`audit_logs`（审计日志）
4. **异常检测**：如果日报中包含高风险内容，自动创建一条异常事件写入 `incidents` 表

### 四、Manager 汇总分析

1. **经理查看日报**：李经理登录系统，调用 `GET /api/v1/report/staff` 查看部门所有员工日报
2. **Manager Agent 聚合**：L2 AI Agent 对所有日报做聚合分析：
   - 整体部门健康度打灯（绿 / 黄 / 红）
   - 提取 Top 3 风险
   - 计算团队效能指数
3. **提交汇总报告**：经理审阅后，调用 `POST /api/v1/report/manager` 提交，写入 `summaries` 表及 `summary_source_links`

### 五、Dream 决策（AI 核心环节）

1. **Dream 引擎启动**：老板陈总调用 `POST /internal/dream/run`，触发 L3 AI Agent
2. **生成 A/B 方案**：Dream 引擎根据经理汇总，生成两个对比决策方案，例如：
   - **方案 A（保守）**：优先补齐交付周期说明文档，稳妥推进，本周不接新客户
   - **方案 B（进取）**：边跟新客户边补文档，加急招聘一个交付专员
3. **附带任务清单**：每个方案下都挂载了建议指派的 `task_candidates`
4. **老板选择**：在 Executive Workspace 看到两个方案的卡片，二选一

### 六、决策落库与任务下发

老板选了方案后，调用 `POST /api/v1/decision/commit`：

1. 决策正式记录到 `decision_records`
2. 方案里的任务候选自动变成正式任务 (`tasks` 表)
3. 任务分配到具体执行人 (`task_assignments` 表)
4. 进入任务队列 (`task_queue` 表)

### 七、闭环——执行回到日报

员工通过 `PATCH /api/v1/tasks/{task_id}` 更新任务状态（接单 → 进行中 → 完成），每次更新写入 `task_updates` 和审计日志。

到了第二天，员工写新日报时看到自己完成的/进行中的任务，这些信息又作为新日报的一部分进入下一轮循环。

```
日报 → 汇总 → AI决策 → 任务分配 → 执行反馈 → 新一轮日报
```

## 前端页面一览

10 个页面，对应侧边栏不同入口：

| 页面 | 目标用户 | 功能 |
|------|----------|------|
| Dashboard | 所有人 | 总览仪表盘（健康检查、任务饼图、审计时间线） |
| Staff Workspace | 员工 | 写日报、看自己的任务、预览提交 |
| Manager Workspace | 经理 | 看部门报告表、撰写汇总、风险面板 |
| Executive Workspace | 老板 | Dream 决策面板、A/B 方案卡片 |
| Task Center | 所有人 | 任务看板，拖拽/更新状态 |
| Incident Center | 经理+ | 异常事件看板，查看/处理异常 |
| Data Flow | 所有人 | Workflow DAG 可视化，展示 21 个节点的实时状态 |
| Agent Console | 开发/管理员 | Agent Adapter 调试面板 |
| API/DB Monitor | 开发/管理员 | API 调用日志、数据库健康状态 |
| Audit | 老板 | 所有写操作的审计日志（谁、什么时候、做了什么） |

右上角有**身份切换器**，点击可在三个身份间切换。切换后：
- API 请求头自动注入新身份的 `X-Actor-Role` / `X-Actor-User-Id` / `X-Node-Id`
- 侧边栏菜单随权限变化
- 数据查询自动按 RBAC scope 过滤

## Demo 模式与真实写入

前端的 `VITE_AUTOMAGE_DEMO_MODE` 是关键开关：

- **Demo 模式 ON**（默认）：不调用后端 API，所有数据来自 `src/data/` 下的 mock 文件，写操作也只是前端模拟
- **Demo 模式 OFF**：真实调用后端 API
- **二次保护**：即使 Demo 模式关闭，写操作还需要 `VITE_AUTOMAGE_ENABLE_REAL_WRITE=true` 才会真正落库

这个设计让你可以先看效果，确认无误后再连真实后端。

## 后端技术骨架

- **框架**：Python FastAPI + SQLAlchemy 2.0 + PostgreSQL + Redis
- **三层保护**：RequestTracking（分配 request_id） → AbuseProtection（限流 + 幂等） → RBAC（角色权限校验）
- **配置**：全部从环境变量（`AUTOMAGE_` 前缀）或 TOML 文件读取，不硬编码
- **审计**：所有写操作自动记录到 `audit_logs` 表
