### block_p0403_b004

当前MVP 设计采用通用Schema 作为主线，包括Staff Schema、Manager Schema、Ex-

### block_p0403_b005

ecutive Schema、Task Schema、Incident Schema 等。这样做的好处是开发链路统一，Agent、

### block_p0403_b006

API、数据库和前端可以围绕一套稳定结构联调，不需要一开始就为不同行业分别设计字段。

### block_p0403_b007

但风险也很明显：如果Schema 过于通用，业务表达会变弱。员工填报时可能觉得字段不

### block_p0403_b008

贴合自己的岗位，Manager 汇总时也可能只能生成比较宽泛的部门摘要，难以体现具体行业

### block_p0403_b009

或岗位差异。

### block_p0403_b010

例如，研发岗位关心接口、代码、Bug、联调、DDL；销售岗位关心客户、商机、跟进阶

### block_p0403_b011

段、成交概率；客服岗位关心工单、满意度、投诉风险。如果全部用一套字段表达，MVP 可

### block_p0403_b012

以跑通，但长期会影响使用体验和数据分析价值。

### block_p0403_b013

MVP 阶段建议采用以下策略：

### block_p0403_b014

先通用Schema，后行业模板Schema。

### block_p0403_b015

也就是说，当前版本先保证所有岗位都能用同一套基础结构表达：

### block_p0403_b016

1. 今天做了什么。

### block_p0403_b017

2. 遇到什么问题。

### block_p0403_b018

3. 是否需要支持。

### block_p0403_b019

4. 明天计划做什么。

### block_p0403_b020

5. 是否存在风险。

### block_p0403_b021

6. 有哪些产出物。

### block_p0403_b022

7. 是否需要上级决策。

### block_p0403_b023

后续再在通用Schema 外层增加行业模板或岗位模板，例如：

### block_p0403_b024

模板类型示例

### block_p0403_b025

### block_p0403_b026

销售岗位模板客户、商机、报价、合同、回款风险

### block_p0403_b027

客服岗位模板工单、投诉、满意度、升级处理

### block_p0403_b028

运营岗位模板活动、数据、投放、转化、异常

### block_p0403_b029

管理岗位模板团队状态、风险、资源、决策事项

### block_p0403_b030

建议本阶段不要为了行业适配过早拆分大量Schema。否则会导致接口、数据库、Agent

### block_p0403_b031

Prompt 和前端表单快速复杂化，影响MVP 主链路交付。

### block_p0403_b032

本问题的推荐结论：

### block_p0403_b033

### block_p0404_b001

### block_p0404_b002

19.2员工填报真实性问题

### block_p0404_b003

员工填报真实性是AutoMage-2 进入真实企业场景后必须面对的问题。

### block_p0404_b004

Staff Agent 可以帮助员工整理日报，但不能保证员工输入一定真实。如果员工随便填写、

### block_p0404_b005

漏填、过度美化，系统后续的Manager 汇总、Executive 决策和Dream 分析都会受到影响。

### block_p0404_b006

这个问题不能只靠AI 解决，也不能简单通过“监控员工”解决。MVP 阶段应采用低侵

### block_p0404_b007

入、可解释、可追溯的方式提高数据可信度。

### block_p0404_b008

主要风险包括：

### block_p0404_b009

1. 员工填写过于简单，例如“正常推进”。

### block_p0404_b010

2. 员工不填写真实阻塞，导致风险无法提前暴露。

### block_p0404_b011

3. 员工夸大完成情况，任务状态与实际不一致。

### block_p0404_b012

4. 员工忘记填写，导致Manager 汇总数据缺失。

### block_p0404_b013

5. Agent 过度润色，使原始问题看起来不明显。

### block_p0404_b014

6. 员工认为系统是考核工具，从而不愿真实表达问题。

### block_p0404_b015

MVP 阶段建议采用以下处理方式：

### block_p0404_b016

19.2.1强制结构化，不强制复杂化

### block_p0404_b017

员工不需要写长篇日报，但必须填写几个关键字段：

### block_p0404_b018

1. 今日完成事项。

### block_p0404_b019

2. 遇到的问题。

### block_p0404_b020

3. 是否需要支持。

### block_p0404_b021

4. 明日计划。

### block_p0404_b022

5. 风险等级。

### block_p0404_b023

6. 确认提交。

### block_p0404_b024

如果员工只输入“今天正常”，Staff Agent 应追问：

### block_p0404_b025

请至少补充一项今日完成事项，并说明明天计划继续推进什么。

### block_p0404_b026

19.2.2员工确认后再入库

### block_p0404_b027

Agent 生成的Staff Schema 必须经过员工确认。未确认内容只能是草稿，不能进入Man-

### block_p0404_b028

ager 汇总。

### block_p0404_b029

这样可以明确责任边界：

### block_p0404_b030

Agent 负责整理；

### block_p0404_b031

员工负责确认。

### block_p0405_b007

### block_p0405_b008

### block_p0405_b009

### block_p0405_b010

### block_p0405_b011

3. missing_staff_ids

### block_p0405_b012

### block_p0405_b013

### block_p0405_b014

### block_p0405_b015

MVP 阶段不建议一开始引入屏幕监控、键盘记录、自动截图等强感知能力。这会显著增

### block_p0405_b016

加员工抵触，也会带来隐私、合规和解释成本。

### block_p0405_b017

推荐先用“员工主动确认+ 任务状态回流+ Manager 审核+ 审计追踪”的方式提高可

### block_p0405_b018

信度。

### block_p0405_b019

本问题的推荐结论：

### block_p0405_b020

MVP 阶段不追求完全自动验证真实性；

### block_p0405_b021

先通过结构化填报、员工确认、任务交叉校验和Manager 审核提高可信度。

### block_p0405_b022

19.3屏幕记忆方案是否进入MVP

### block_p0405_b023

屏幕记忆指系统通过截图、窗口记录、操作轨迹、文档访问记录等方式，自动采集员工

### block_p0405_b024

工作过程，再由Agent 整理为工作记忆或日报内容。

### block_p0405_b025

这个方向长期有价值，因为它能减少员工填报负担，也能让系统更接近真实工作现场。但

### block_p0405_b026

它不适合过早进入MVP 主链路。

### block_p0405_b027

原因有四点。

### block_p0405_b028

第一，技术复杂度高。屏幕记忆涉及客户端采集、权限授权、数据压缩、OCR、多模态

### block_p0405_b029

理解、隐私过滤、存储成本和检索能力，远超当前MVP 的核心验证范围。

### block_p0405_b030

第二，隐私和合规风险高。屏幕可能包含客户信息、个人聊天、密码、财务数据、合同内

### block_p0405_b031

容等敏感信息。如果没有完善的本地过滤和权限控制，企业很难接受。

### block_p0405_b032

第三，员工心理阻力大。如果MVP 一开始就强调屏幕采集，员工很容易把系统理解为

### block_p0405_b033

监控工具，而不是协作工具。

第四，它会分散主链路。AutoMage-2 当前最需要验证的是：
